## 一、什么是 Vibe Coding？

Vibe Coding 是一种以 AI 为执行主体的开发方式：开发者用自然语言描述"要什么"，AI 负责"怎么写"。但**绝不是甩手掌柜**——人类依然掌控**方向、质量、验证**三个关键环节。

本项目就是一个典型案例：用 AI 辅助从零搭建了一个完整的无线通信仿真系统（9 个 Python 模块 + 4 份文档 + 22 个测试用例全部通过），全程没有手写一行核心逻辑代码，但人类在**理解需求、发现 bug、设计决策**上发挥了不可替代的作用。

> 💡 关键认知：AI 把"写代码"变得极廉价，于是**代码不再是瓶颈，规约和验证才是**。Vibe Coding 的真问题不是"怎么让 AI 写代码"，而是"怎么保证 AI 写的是对的"——三条方法论就是来回答这个问题的。

## 二、方法论速查：从 Vibe-Spec-Harness 到 SDD / Mock / TDD + spec

Vibe Coding 听起来像"凭感觉写"，但它有完整工程框架做骨架。老师 PDF 前半段把这个框架抽象成 **Vibe → Spec → Harness** 三层演进，本工作流把它落地为 **SDD / Mock / TDD + spec** 三条方法论 + 一个核心概念（spec=DESIGN.md）。

### 2.1 Vibe-Spec-Harness 三层框架

这是老师 PDF 第 5-9 页的核心观点：AI Agent 时代，开发的核心矛盾从"代码写得够不够快"转变为 **"方向是否被探索、共识是否被冻结、结果是否被验证"**。

| 层级 | 名字 | 作用 | 要回答的问题 | 输出物 |
|------|------|------|-------------|--------|
| **Vibe** | 探索/试错 | 用最低成本获取有效认知 | 做什么产品、有什么功能、成功标准是什么 | PRD |
| **Spec** | 规格收敛 | 把隐含判断变成显式、可评审的工程约束 | 做什么/不做什么、边界、输入输出、状态流转、异常处理、什么算完成 | design-spec.md / DESIGN.md |
| **Harness** | 约束验证 | 让 spec 可被执行、检查、验证、回归和治理 | Agent 看到什么上下文、能调用什么工具、结果如何验证 | 测试 + 工具约束 + CI + 评审 |

三层关系：

```
Vibe（探索） → Spec（冻结共识） → Harness（约束结果）
   brainstorming      writing-plans       TDD + Subagent + Verification
   输出 PRD           输出 DESIGN.md      输出可验证、可回归的代码
```

> 🔑 **关键认知**：Vibe 只应发生在前期；一旦进入 Spec 阶段，就要停止自由探索，把共识钉死；进入 Harness 阶段后，**验证是执行的完成条件**——没通过验证的代码不能合并交付。

### 2.2 框架到落地的对照：Vibe-Spec-Harness ↔ SDD/Mock/TDD ↔ 七阶段

把三层框架、三条方法论、七阶段工作流叠在一起看，就不会混：

| 框架层 | 落地方法论 | 对应七阶段 | 核心产出 | 人类角色 |
|--------|-----------|-----------|----------|----------|
| **Vibe** | 需求探索 / Brainstorming | 阶段 1 Understand | PRD / 硬约束清单 | 提意图、审理解 |
| **Spec** | **SDD**（Spec-Driven Development） | 阶段 2 Plan | DESIGN.md | 审批规约 |
| **Harness** | **Mock** + **TDD** + 验证约束 | 阶段 3 Mock + 阶段 4 Build + 阶段 5 Debug + 阶段 6 Verify | 测试、MOCK_TEST_REPORT、可运行代码 | 判断设计是否自洽、诊断根因、验收 |

**一句话串起来**：
- Vibe 阶段用 **brainstorming** 出 PRD；
- Spec 阶段用 **SDD** 把 PRD 写成 DESIGN.md；
- Harness 阶段用 **Mock** 验设计、用 **TDD** 写代码、用测试/评审/CI 保证可回归。

> 💡 **为什么需要这个对照？** 老师 PDF 讲的是抽象框架（Vibe-Spec-Harness），本工作流讲的是可执行流程（七阶段），而 SDD/Mock/TDD 是两者之间的方法论桥梁。对照表让读者看到“抽象概念”怎么变成“今天要干什么”。

### 2.3 Spec 详解：6 个问题 + spec 是什么

PDF 第 8 页强调：Spec 不是"写文档"，而是建立团队可对齐的正式语义层。一份合格的 Spec 至少回答：

1. **做什么 / 不做什么** —— 范围边界
2. **边界在哪里** —— 模块职责划分
3. **输入输出是什么** —— 接口契约
4. **状态如何流转** —— 数据流与处理链路
5. **异常如何处理** —— 错误路径与兜底
6. **什么算完成** —— 验收标准

本项目的 `DESIGN.md` 就是这 6 个问题的落地：系统链路回答 1/2/4，模块设计回答 3，预期风险回答 5，运行方式+验收标准回答 6。

**spec（规约）= DESIGN.md**——一份精确描述"系统该做什么"的文档：架构、模块接口、算法选择、关键参数、预期风险。它不是某个方法论的缩写，而是**整个流程唯一对齐基准**：

- SDD 围绕它：先写 spec，再写代码
- Mock 验证它：用模拟手段验 spec 是否自洽
- TDD 落实它：把 spec 翻译成测试
- 收尾对照它：代码/测试/报告都回溯 spec

> 📌 **文档不是写完代码补的装饰，而是 spec——代码、测试、报告共同的对齐基准。** 这正是老师强调"先写 DESIGN"的本质：DESIGN 就是 spec，一切以它为准。PDF 中提到的 **"Markdown 即代码"（Markdown-as-Code）** 也是同一个意思：把 PRD → Design → Spec → Plan 每一层都用 Markdown 文档作为单一事实来源，确保需求在传递过程中不丢失、不变形。

### 2.4 Harness 详解：四类约束 + Superpowers + 五步落地

PDF 第 9 页指出，Harness 要解决 **Agent 看到什么上下文、能调用什么工具、结果如何验证**。下面按"约束 → 机制 → 落地步骤"三层展开。

#### 四类约束

| 约束类型 | 作用 | 本项目落地 |
|----------|------|-----------|
| **上下文约束** | 文档入口与知识检索 | PRD + DESIGN.md + 测试文件作为 AI 的上下文边界 |
| **工具约束** | 权限边界与操作白名单 | Claude Code 工具权限、Bash/Write/Edit 的人类确认 |
| **验证约束** | 测试、构建与静态检查 | 22 个公开测试、端到端 diff、metrics.json 字段检查 |
| **治理约束** | 评审、CI 与变更审计 | MOCK_TEST_REPORT.md、AI_LOG.md、版本控制 |

Harness 的目标：**让正确变化更容易发生，让错误变化更早暴露**。

#### Superpowers 三大核心机制

PDF 第 10-14 页提到 Claude Code / Superpowers 的技术本质是 **Markdown-as-Code + 文本驱动的状态机**，并给出三个核心机制：

1. **System Prompt 注入**：把项目级约束（如 `.claude/CLAUDE.md`、`AGENTS.md`）作为系统提示词注入，让 AI 在每次交互里都遵守仓库级规范。
2. **子代理上下文隔离**：复杂任务拆给 Subagent，每个子代理有独立上下文，避免互相干扰（见第九章）。
3. **两阶段代码审查**：先由 AI 自检，再由独立 Verify Subagent 或人类复核，确保实现与 spec 一致。

物理载体是 **Skill 目录**：每个 Skill 包含 `SKILL.md` + Prompt 模板 + 流程图（见 9.3）。

#### 五步建立工程级 Agent 开发体系

PDF 第 16 页给出最小落地路径，可直接复用：

1. **建立统一 Agent 入口**：写 `AGENTS.md`，规定团队协作规范、仓库级开发约束、工具使用规则。
2. **补最小 Spec 骨架**：让项目先达到 SDD 级别——需求 / 设计 / 契约 / 验收四要素齐全。
3. **先做高价值 Harness**：优先覆盖高频改动路径、高风险链路、跨模块边界、曾出过事故的行为。
4. **把问题沉淀成资产**：每次真实 bug 都落成 **Spec Update** 或 **Harness Update**，避免重复踩坑。
5. **形成季度级回顾**：检查哪些规格长期失效、哪些约束长期不跑、哪些任务仍然高风险。

> 💡 本项目如果能做到第 4 步，QPSK 解调规则错误和前导伪峰值这两个 bug 就会被写进"预期风险"和"Mock 验证清单"，下次不会再犯。

### 2.5 SDD / Mock / TDD 三条方法论

#### 一句话版

| 方法论 | 全称 | 一句话 | 核心动作 | 对应你项目的什么 |
|--------|------|--------|----------|------------------|
| **SDD** | Spec-Driven Development 规约驱动开发 | 先写 spec，再写代码；spec 是唯一信源 | DESIGN.md 就是 spec，代码/测试/报告都对照它 | `DESIGN.md`（架构/接口/参数/风险） |
| **Mock** | 模拟/打桩验证 | 用假的、可控的对象提前验证 spec | 手推 / 最小脚本 / 桩函数，验证还没实现的模块 | 写代码前算前导自相关、手推 QPSK 映射闭环 |
| **TDD** | Test-Driven Development 测试驱动开发 | 先写测试，再写实现 | 红（测试失败）→绿（写代码通过）→重构 | 老师给的 20 个公开测试就是天然起点 |

#### 怎么串成一条线

```
spec ──── 规约(DESIGN)，定义"做什么"，是唯一信源
  │
  ▼
SDD ──── 先写 spec，再写代码（规约驱动）
  │
  ▼
Mock ─── 用模拟手段验证 spec 是否自洽(验设计，不验代码)
  │
  ▼
TDD ──── 把 spec 落成测试，定义"什么算对"(先红后绿)
  │
  ▼
AI 生成代码 ── 照着"已验证的 spec + 已写好的测试"去实现
  │
  ▼
对照 spec 收尾 ── 代码/测试/报告回溯对照规约，保证一致
```

**一句话**：spec 是信源，SDD 定方向，Mock 验设计，TDD 定标准——四者给"AI 写代码"画了跑道，AI 在跑道里跑，人类在跑道外掌方向盘。

#### 为什么 AI 时代突然重要

| 方法论 | 前 AI 时代 | AI 时代 |
|--------|---------|--------|
| spec | 规约是人写给人看的 | 规约是**人写给 AI 看的施工图**，也是**对抗 AI 自由发挥的锚** |
| SDD | 规约驱动实现 | AI 照 spec 生成代码比人手写高效得多 |
| Mock | 验证设计可行性 | **在 AI 生成大量代码前，先廉价验证 spec**，避免生成一堆有设计缺陷的代码 |
| TDD | 测试是质量保障 | 测试是**AI 的行为规约**——测试定义"什么算对"，AI 去满足它，自动可验证 |

#### 本项目对照：我们做对/做错了什么

老实说，我们这次开发**部分违背**了这些方法论，踩了本可避免的坑：

| 方法论 | 规范做法 | 我们实际做的 | 后果 |
|--------|---------|-------------|------|
| SDD | 先写 spec，AI 照 spec 实现；收尾对照 spec | 写了 DESIGN 但不够细，AI 实现时仍有自由发挥 | — |
| Mock | 写代码前用手推/小脚本验 spec | **跳过了**，直接写代码 | QPSK 解调规则、前导伪峰值两个 bug 等到运行时才发现 |
| TDD | 先看公开测试（红），再写代码（绿） | 先写完代码再跑测试 | 顺序反了，bug 没被测试提前拦截 |

**教训**：如果我们严格按 spec→SDD→Mock→TDD 走，那两个 bug 在写代码**前**就该被 mock 消灭。本工作流的后续章节，就是把这个规范流程沉淀下来，避免重蹈覆辙。

## 三、完整工作流总览（七阶段）

严格对应 PRD §8"工程流程要求"——**不得直接跳到最终代码生成**。

```
┌─────────────────────────────────────────────────────────────┐
│  阶段 0：环境与依赖准备                                       │
│  创建隔离环境 → 安装依赖 → 确认工具链可用                     │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 1：需求探索（Understand）              【为 SDD 喂料】  │
│  AI 并行探索 PRD/代码/测试 → 输出结构化理解                   │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 2：方案规划（Plan）                  【SDD：写规约】    │
│  写 DESIGN.md → 架构/接口/算法/参数/预期风险 → 人类审批       │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 3：Mock 验证设计（Verify Spec）  【Mock：验规约】 ★    │
│  手推/最小脚本/桩函数 → 验接口/帧结构/同步/端到端是否自洽     │
│  发现问题 → 回阶段 2 修订 DESIGN，写 MOCK_TEST_REPORT，再验   │
│  ────── DESIGN 稳定后才进阶段 4 ──────                       │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────┐   ┌──────────────────────────┐
│  阶段 4：并行实现（Build）   │◀──│  人类确认细节/澄清歧义    │
│  AI 按"已验证规约"批量生成   │   │  【TDD：测试先于/伴随代码】│
└──────────────┬──────────────┘   └──────────────────────────┘
               ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 5：运行调试（Run & Debug）   ◀─── 失败则回到实现       │
│  跑端到端 → 发现 bug → 人类定位根因 → AI 修复                 │
└──────────────────────────┬──────────────────────────────────┘
                           ▼
┌─────────────────────────────────────────────────────────────┐
│  阶段 6：验证收尾（Verify）            【对照 spec 收尾】  │
│  跑测试套件 → 人工复核输出 → 代码/测试/报告回溯对照 DESIGN    │
└─────────────────────────────────────────────────────────────┘
```

### 与老师 SwiftClaw 五步法对照

PDF 中项目一 SwiftClaw 把工程流程抽象为 5 个连续动作，和本七阶段工作流本质一致：

```
Brainstorming → Writing Plans → Git Worktree → TDD 开发 → Verification
   需求澄清        生成 Spec       物理隔离沙箱     红绿重构      代码审查/Subagent 并行
```

对照映射：

| SwiftClaw 五步 | 本工作流七阶段 | 关键动作 |
|----------------|---------------|----------|
| Brainstorming | 阶段 1 Understand | 与 AI 多次交互，明确需求、成功标准 |
| Writing Plans | 阶段 2 Plan + 阶段 3 Mock | 出 DESIGN.md，验证后再稳定 |
| Git Worktree | 阶段 4 Build 的前提 | 用物理隔离沙箱防止污染主分支 |
| TDD 开发 | 阶段 4 Build | 先红后绿，测试定义正确 |
| Verification | 阶段 5 Debug + 阶段 6 Verify | Subagent 并行审查，对照 spec 收尾 |

SwiftClaw 的核心理念——**“先结构，后代码”**——与本工作流完全相同：一切动作围绕“目的”展开，用 SDD 确保 AI 与人达成共识，用 TDD 确保代码质量，用 Git Worktree 实现物理隔离。

### 从意图到交付的完整闭环

PDF 第 15 页给出的演进链路可以映射到本七阶段工作流：

```
工程意图 / 需求目标 ──→ 规格收敛 ──→ 约束验证 ──→ 下一轮持续演进
    Intent                 Spec           Validate              Next
      │                      │                │                  │
      ▼                      ▼                ▼                  ▼
   Vibe(阶段1)        Plan+Mock(阶段2-3)  Build+Debug+Verify  更新 Spec/Harness
  brainstorming        冻结 DESIGN 共识    (阶段4-6)           通过 → 合并交付
   探索试错           writing-plans      TDD + Subagent       失败 → 更新 Spec/Harness
                                          执行实现 + 验证
```

**三条原则**：
1. **探索被明确限定在前期**——阶段 1 之后不再自由 vibe。
2. **验证是执行完成条件**——阶段 6 不通过，代码不能视为完成。
3. **反馈是下一轮输入**——每次失败都更新 spec 或 harness，沉淀为资产。

> 🔑 **灵魂约束**：阶段 4（写代码）必须在阶段 3（DESIGN 验证稳定）之后。这是整个流程的核心，也是老师强调"先 design 再对 design 验证"的落点。

## 四、各阶段详解（以本项目为例）

### 阶段 0：环境与依赖准备 🛠️

**做什么**：在写任何代码前，先把"工地"搭好。

**本项目实践**：
```bash
conda create -n wireless-final python=3.11 -y   # 隔离环境
```

### 阶段 2：方案规划（Plan）📋  ·  方法论：SDD（规约驱动）

**做什么**：在动手前，把"要建什么"写下来。这是 Vibe Coding 中**人类控制权最重要的一环**，也是 SDD 的核心——**规约先行**。

**为什么这步是 SDD**：SDD 要求"先有规约，再有实现"。在 AI 时代，规约（DESIGN.md）就是写给 AI 的施工图。规约越精确，AI 实现越不容易跑偏；规约越模糊，AI 自由发挥空间越大，返工风险越高。**写规约是人类把意图"钉死"的过程，AI 无法替代。**

**DESIGN.md 该写什么（6 个部分，模块设计与参数可并入系统链路）**：

| # | 部分 | 作用 | 本项目示例 |
|---|------|------|-----------|
| 1 | 系统概述 | 一句话讲清做什么 + 背景 | 无线通信基带仿真，文本→信道→文本 |
| 2 | 系统链路 + 模块设计 + 关键参数 | 数据流图，逐环节；每模块给职责/接口/算法/参数；全局参数集中表 | Source Encode→…→Source Decode；`qpsk_modulate(bits:list)→np.ndarray`；snr=12, seed=2026 |
| 3 | **数据流推演** | 比特数变化链，收发对齐验证 | 6128bit→×3→+82→÷2→9233 符号 |
| 4 | **预期风险** ★ | 提前列设计风险 + 现象 + 根因 + 缓解 | 周期前导自相关多峰→换伪随机 |
| 5 | 文件结构 | 目录树 | src/ 下 8 个模块 |
| 6 | 运行方式 + 验收标准 | CLI 命令 + 通过门限 | SNR≥12dB 文本完全一致 |

> 💡 也可以把模块设计、关键参数拆成独立章节（变成 8 部分）， whichever 更清晰。核心不是章节数，而是**接口、参数、数据流、风险**四项必须写细。

> 💡 **#5 数据流推演 和 #6 预期风险 是 DESIGN 的灵魂**，也最容易被跳过。#5 是 mock 验证端到端的基础，#6 是 mock 验证的设计假设来源。我们的 DESIGN.md 当初**缺了这两部分**，所以 QPSK 解调规则、前导伪峰值两个 bug 毫无防备地拖到运行时才暴露。

**怎么驱动 AI 写 DESIGN（分步法，别一句"帮我写 DESIGN"丢给 AI）**：

核心认知：让 AI 从"理解 PRD"逐步收敛到"产出规约"，每步人类都审。

**第 1 步：先让 AI 读 PRD、提取硬约束（不写 DESIGN）**
```
读 PRD.docx，提取以下硬约束，不要写 DESIGN：
1. 固定链路顺序（逐环节列出）  2. 统一验收入口（CLI+输出文件）
3. 必做 vs 可选（基础必做 QPSK/AWGN）  4. metrics.json 最低字段

```
→ 人类审：约束提取对不对。这是后续一切锚点。

**第 2 步：让 AI 先出 DESIGN 骨架（只列 6-8 部分大纲，不填内容）**
```
基于硬约束，先只给 DESIGN.md 的大纲（6-8 部分），
每部分一句话说明要写什么。先别填具体内容，等我确认骨架。
```
→ 人类审：系统链路、模块接口、数据流推演、预期风险、验收标准是否都有？**这一步防止 AI 写偏。**

**第 3 步：逐章驱动 AI 填充（每章定向提问，逼它写细）**
| 章节 | 驱动 prompt 关键词 |
|------|-------------------|
| 模块设计 | "每个模块给**函数签名**（参数类型+返回类型）、算法选择理由、关键参数" |
| 关键参数 | "全局参数集中成表：名/默认值/取值理由" |
| 数据流推演 | "推演 262 字符文本到 QPSK 符号数的**比特数变化链**，标每步乘除加" |
| 预期风险 | "列 5 个可能在 mock/运行时暴露的设计风险，每个写：现象+根因+缓解" |

**第 4 步：人类主导补"预期风险"（这一节 AI 容易写得泛）**
基于领域知识补具体风险，本项目就该写：
- "前导序列若用周期性，自相关会有伪峰值，同步失效" ← 实际踩过
- "QPSK 解调 b0/b1 判决若搞反实部虚部，无噪声下也全错" ← 实际踩过

> 这两个风险若写进 DESIGN，mock 阶段会主动去验，根本不会拖到运行时。这正是"先 design 再对 design 验证"的落点。

**为什么重要**：
- DESIGN.md 是**人类审批 AI 的载体**——你看到 AI 打算怎么建，不同意就改
- 它是后续 Mock 验证、TDD 测试、代码实现的**共同基准**（规约是唯一信源）
- 写规约的过程逼着人类想清楚架构，这是 AI 替代不了的

**人类动作**：审批 DESIGN、澄清歧义（用 AskUserQuestion）、补充约束。

### 阶段 3：Mock 验证设计（Verify Spec）🔬  ·  方法论：Mock  ★ 老师强调的核心

**做什么**：DESIGN 写完后、代码写之前，用**最廉价的模拟手段**验证规约是否自洽。这一步最容易跳过，也最值钱。

**为什么这步是 Mock**：Mock 的本质是"用假的、可控的对象替掉真实依赖来验证"。在写代码前，真实模块还不存在，但可以用**手推 / 最小脚本 / 桩函数**三种 mock 手段，提前证明设计假设成立。**Mock 测的是设计，不是代码**——这是它和 TDD（测代码）的关键区别。

**Mock 验证四个维度**（对应 PRD §8"验接口、帧结构、同步、端到端"）：

| 维度 | 验什么 | Mock 手段 |
|------|--------|-----------|
| 接口 | 模块间数据类型对得上吗 | 列函数签名，画数据流图，检查"上一步输出能否当下一步输入" |
| 帧结构 | 字段长度/偏移/可逆性算得清吗 | 手算帧总长，手推 build→parse 能否还原 |
| 同步 | 前导选得对吗、能定位吗 | **3 行脚本算前导自相关**，看是不是尖锐单峰 |
| 端到端 | 整条链路逻辑闭环吗 | 推演比特数变化链，检查收发两端能否对齐 |

**三种 Mock 手段详解**：

- **手段 A：手推/手算**（最廉价，验算法选择）。例：手推 QPSK 4 种比特对的调制→解调闭环，就能发现解调规则对不对。
- **手段 B：最小验证脚本**（验数据流和接口）。例：生成候选前导，算它的自相关，看是不是尖锐单峰——能发现伪峰值。不写完整模块，只验一个假设。
- **手段 C：桩函数**（验接口契约）。每个模块先写空壳 + docstring，串起来跑通调用链（哪怕返回假数据），证明接口自洽。

**发现 mock 问题怎么办**：回到阶段 2 修订 DESIGN，写进 `MOCK_TEST_REPORT.md`，再 mock 验证，直到稳定。**DESIGN 稳定后才进阶段 4 写代码。**

> 💡 本项目我们跳过了这步，吃了两个亏——详见[第六章反例](#六用本项目的两个真实-bug-作反例)。

**人类动作**：主导 mock 验证（AI 写脚本，人类判断结果是否符合预期、是否有设计缺陷）。这是 SDD→TDD 之间人类参与度最高的一步。

### 阶段 4：并行实现（Build）⚡  ·  方法论：TDD（测试驱动）

**做什么**：AI 按"已验证的规约"批量生成代码和文档。

**为什么这步关联 TDD**：规范做法是**先看公开测试（让它失败=红），再让 AI 写代码通过测试（绿）**。本项目的公开测试（TC-T-004~017）就是天然的 TDD 起点——它们定义了"什么算对"。先确认测试能跑且失败，再写实现，bug 能被测试提前拦截。我们这次顺序反了（先写代码再跑测试），所以两个 bug 没被提前拦住。

**关键技巧：独立文件并行生成，有依赖的串行生成。**

**本项目实践**：
- **并行生成**（8 个模块互不依赖，一条消息里 8 个 Write 调用）：
  `source.py`、`crypto.py`、`channel_coding.py`、`modulation.py`、`channel.py`、`framing.py`、`synchronization.py`、`metrics.py`
- **并行生成**（文档互不依赖）：
  `main.py`、`DESIGN.md`、`TEST_PLAN.md`、`MOCK_TEST_REPORT.md`、`AI_LOG.md`

**Git Worktree 物理隔离沙箱**：

SwiftClaw 强调用 Git Worktree 做“物理隔离沙箱”——每个 phase / 每个 Subagent 在独立工作目录开发，防止污染主分支：

```bash
# 为主分支创建一个独立工作树，用于 AI 实验
mkdir -p ../wireless-final-build
cd ..
git worktree add wireless-final-build -b feature/ai-impl

# AI 在这个目录里生成/修改代码
# 人类审完后再 merge 回主仓库
cd wireless-final-build
# ... AI 生成代码 ...
```

在 Claude Code 里更简单的等价做法：对 Agent 调用时加 `isolation: "worktree"`，系统会自动创建临时工作树，改动不会污染当前分支。

**为什么重要**：并行生成把"串行写多个文件"的时间压缩到一次往返，大幅提速；Git Worktree 则把“AI 实验”和“人类主分支”物理隔开，改坏了也能随时丢弃。

**人类动作**：观察生成结果，发现明显问题立即叫停；Worktree 实验通过后再合并回主分支。

### 阶段 5：运行调试（Run & Debug）🐛

**做什么**：这是 Vibe Coding 中**人类价值最高的环节**——AI 写的代码几乎必然有 bug，得人来诊断。

**本项目踩的两个真实坑**：

#### 坑 1：QPSK 解调器判决规则写反
- **现象**：无噪声下 QPSK 调制→解调就有 332 个比特错误（应该 0 个）
- **人类定位**：写了个小测试逐符号打印，发现 `(0,1)` 比特对调制后解调成了 `(1,0)`
- **根因**：AI 把 `b0`/`b1` 到底由实部还是虚部决定搞混了
- **修复**：明确规则——`b0` 看虚部符号，`b1` 看实部符号
- **反思**：这本质是设计 bug，阶段 3 手推映射表就能发现

#### 坑 2：Preamble 伪随机性不足导致同步伪峰值
- **现象**：BER=0.49（几乎是随机猜），同步检测到的起始位置是 608（应该是 0）
- **人类定位**：发现 Preamble 用了周期序列 `[1+1j,-1+1j,...]` 循环，自相关到处是峰
- **根因**：周期性序列的循环移位与自身高度相关，同步找不到真正起点
- **修复**：改用独立种子生成的伪随机 QPSK 符号作 Preamble，自相关变尖锐单峰
- **反思**：这本质是设计 bug，阶段 3 写 3 行自相关脚本就能发现

**调试方法论（可复用）**：
1. **先跑端到端**，看宏观指标对不对（BER、匹配率）
2. **指标不对就缩范围**：单独测每个模块（先测 QPSK mod/demod，再测 frame，再测 sync）
3. **打印中间数据**：让 AI 写一段调试脚本，逐符号/逐比特对比发送和接收
4. **定位根因再修**：不要让 AI 瞎改，人类要先理解"为什么错"

**人类动作**：这是 AI 最帮不上忙的阶段。AI 能写代码，但不擅长**诊断自己写的代码为什么错**。人类的领域知识（懂 QPSK 星座图、懂自相关特性）是关键。

## 六、用本项目的两个真实 bug 作反例

我们这次开发**跳过了阶段 3**（没对 DESIGN 做 mock 验证就直接写代码），结果运行时才暴露两个本质是"设计错误"的 bug。下面用"如果做了 mock 验证会怎样"来对比：

### 反例 1：QPSK 解调规则写反

**实际发生（违背流程）**：
- 直接让 AI 写 `modulation.py`
- 运行端到端，BER=0.49（几乎全错）
- 写调试脚本逐符号打印，才发现 `(0,1)` 调制后解调成了 `(1,0)`
- 根因：解调时 `b0`/`b1` 该看实部还是虚部搞混了

**如果做 mock 验证（规范流程）**：
- 在 DESIGN 阶段，手推一遍映射表：

| 输入比特对 | 调制后复数 | 实部符号 | 虚部符号 | 解调应判回 |
|-----------|-----------|---------|---------|-----------|
| 00 | (1+1j)/√2 | + | + | 00 |
| 01 | (-1+1j)/√2 | − | + | 01 |
| 11 | (-1-1j)/√2 | − | − | 11 |
| 10 | (1-1j)/√2 | + | − | 10 |

- **手推时就会发现问题**：要区分 00 和 10（实部都为 +），只能靠虚部；要区分 00 和 01（虚部都为 +），只能靠实部。于是必然得出"`b0` 看虚部、`b1` 看实部"——**这个结论在设计阶段就该写进 DESIGN，根本不该等到运行时才发现**。
- Mock 验证只花了 5 分钟手推，省下了后面 1 小时的调试。

### 反例 2：Preamble 伪随机性不足导致同步伪峰值

**实际发生（违背流程）**：
- 直接用了周期序列 `[1+1j, -1+1j, -1-1j, 1-1j]` 循环做前导
- 运行后 BER=0.49，同步检测到起始位置 608（应该是 0）
- 调试发现前导自相关到处是峰，根本定位不到帧起点
- 根因：周期序列的循环移位与自身高度相关

**如果做 mock 验证（规范流程）**：
- 在 DESIGN 阶段，写一个 3 行脚本算前导的自相关：
  ```python
  preamble = np.array([1+1j,-1+1j,-1-1j,1-1j]*7)[:25]
  autocorr = np.correlate(preamble, preamble, 'full')
  ```
- **一看输出就发现**：自相关到处是大值，不是尖锐单峰——前导设计不合格，必须换伪随机序列。
- Mock 验证只花了 3 行代码，省下了后面定位"为什么 BER 是 0.49"的痛苦。

### 反例总结

| | 违背流程（实际） | 规范流程（该做） |
|---|---|---|
| 发现 bug 的时机 | 代码运行后 | 写代码前的设计阶段 |
| 发现 bug 的成本 | 跑端到端+写调试脚本+逐符号打印 | 手推映射表 / 3 行自相关脚本 |
| bug 性质 | 以为是"代码 bug" | 本质是"设计 bug" |
| 修复方式 | 瞎改试错 | 改 DESIGN 再改代码 |

**两个 bug 都是设计错误，本该在设计阶段用 mock 消灭。** 这正是老师强调"先 design 再对 design 验证"的原因。

## 七、MOCK_TEST_REPORT.md 怎么写

Mock 验证发现问题后，要形成闭环——**修订 DESIGN + 写报告**。PRD §8 要求 MOCK_TEST_REPORT.md 说明"发现的问题和调整内容"。

> 🔑 **关键认知**：MOCK_TEST_REPORT 不是"测试通过的报告"，而是"发现了设计问题并改进的报告"。**没有发现问题的 mock 报告，要么是设计完美（罕见），要么是 mock 没认真做（常见）。**

结构模板：

```markdown
# MOCK_TEST_REPORT.md

## 1. Mock 测试场景（≥3 个，对应 PRD §8 要验证的维度）
- Mock 1：验证 QPSK 映射闭环
- Mock 2：验证前导自相关特性
- Mock 3：验证帧结构可逆性

## 2. 发现的问题/风险/缺陷（≥1 个）  ← 这才是 mock 的价值
- 问题 1：周期性前导自相关有多峰，同步会失效
  - 影响：BER≈0.5，无法定位帧起点
  - 根因：周期序列循环移位与自身相关

## 3. DESIGN.md 修订记录  ← 闭环的关键
- 修订 1：前导序列由周期序列改为伪随机序列（seed=42 生成）
  - 修订前：[1+1j,-1+1j,...] 循环
  - 修订后：伪随机 QPSK 符号
  - 修订理由：mock 验证发现周期序列自相关有多峰
- 修订 2：DESIGN 新增"预期风险"小节
```

## 八、人类 vs AI 的职责分工

这是 Vibe Coding 的核心认知——**不是所有事都丢给 AI**。按 **Vibe → Spec → Harness** 三层框架来划分职责，每层都有不同的"AI 能做 vs 人类必须做"边界：

| 框架层 | 环节 | AI 负责 | 人类负责 | 主导方法论 |
|--------|------|---------|----------|-----------|
| **Vibe** | 环境搭建 + AGENTS.md | ❌ | ✅ 执行命令、选版本、起草仓库级约束 | — |
| **Vibe** | 需求理解 | ✅ 快速读代码 / PRD 总结 | ✅ 判断理解对不对、提需求 | SDD 喂料 |
| **Spec** | 方案规划（DESIGN.md） | ✅ 起草 / 搜索选项 | ✅ 审批、定架构、做决策 | **SDD** |
| **Harness** | Mock 验证设计 | ✅ 写验证脚本 | ✅ **判断设计是否自洽** | **Mock** |
| **Harness** | 代码实现 | ✅ 批量生成（按 spec） | ✅ 观察、叫停明显错误 | TDD |
| **Harness** | 调试定位 | ⚠️ 能写调试脚本 | ✅ **诊断根因（最关键）** | — |
| **Harness** | 验证收尾 | ✅ 跑测试 / 生成图表 | ✅ 判断结果合理性、对照 spec | **对照 spec** |
| **Harness** | 文档 | ✅ 起草 | ✅ 审阅、补充思考 | SDD |
| **Harness** | Subagent 编排 | ✅ 执行子任务 | ✅ 决定拆几个 / 拆在哪 | 2.2 框架 + 9.x Subagent |

**一句话总结**：AI 是高效的"执行者"和"起草者"，人类是不可替代的"决策者"和"诊断者"——Vibe 阶段定方向，Spec 阶段定边界，Harness 阶段定验收。
| 调试定位 | ⚠️ 能写调试脚本 | ✅ **诊断根因（最关键）** | — |
| 验证收尾 | ✅ 跑测试 | ✅ 判断结果合理性、对照 spec | **对照 spec** |
| 文档 | ✅ 起草 | ✅ 审阅、补充思考 | SDD |

**一句话总结**：AI 是高效的"执行者"和"起草者"，人类是不可替代的"决策者"和"诊断者"。

## 九、Subagent 拆分与 Skills 蒸馏

Subagent 既是老师 PDF 前半段 Superpowers 的核心机制 ②（子代理上下文隔离），也是后半段 SwiftClaw / TA-Claw 的工程实践。两者结合起来，回答了一个真问题：**单个 AI 会话的上下文有限，把大任务拆小才能既快又稳**。

### 9.1 为什么要拆 Subagent？

| 问题 | 不拆的后果 | 拆分后的好处 |
|------|-----------|-------------|
| 上下文爆炸 | AI 同时记代码、记 PRD、记设计，容易丢细节 | 每个子代理只负责一个狭窄目标 |
| 串行太慢 | 一件事做完才能做下一件 | 独立任务并行跑， wall-clock 大幅缩短 |
| 验证不独立 | 自己写代码自己验，容易盲区 | 用独立 Verify subagent 审设计/审代码 |
| 能力混杂 | 一个 Agent 既做架构又做实现又做测试 | 不同 Agent 承担不同角色，边界清晰 |

### 9.2 拆分的三个维度

**维度 1：按功能模块拆（TA-Claw 模式）**

把产品拆成几条独立的功能线，每条线一个 Subagent/Skill：

```
TA-Claw
├── F1 智能课件生成（ta-courseware）
├── F2 学生答疑 Bot（ta-answer）
└── F3 实验运营管理（ta-experiment）
```

每个模块有独立输入、独立输出、独立验证标准，开发时可以并行推进。

**维度 2：按工作流阶段拆（SwiftClaw 模式）**

```
Brainstorming → Writing Plans → Git Worktree → TDD → Verification
   需求澄清        生成 Spec       物理隔离沙箱     红绿重构     代码审查
```

不同阶段交给不同能力的 Agent：
- **Explore / 探索型**：读 PRD、读代码、总结约束（只读不写）
- **Plan / 规划型**：出架构、出接口、出 DESIGN.md
- **Build / 实现型**：按已验证 spec 写代码
- **Verify / 审查型**：跑测试、审代码、找设计缺陷

**维度 3：按独立文件/模块并行生成（本项目实际做法）**

当设计已稳定，8 个源文件互不依赖，可以一次性让 AI 并行写出：

```
source.py  crypto.py  channel_coding.py  framing.py
modulation.py  channel.py  synchronization.py  metrics.py
```

> 💡 关键前提：**必须先完成阶段 2（Plan/Spec）和阶段 3（Mock 验证）**。没有稳定设计就并行生成，会同时产生一堆有设计缺陷的代码，返工成本更高。

### 9.3 Skills 蒸馏：把子任务沉淀为可复用能力

TA-Claw 的另一条经验是：不要每次从零写 prompt，而是把高频任务封装成 **Skill**。

一个 Skill 的标准结构：

```
skill-name/
├── SKILL.md          # 该 Skill 的职责、输入输出、流程图
├── prompt.md         # 可复用的 prompt 模板
└── flowchart.png     # 人类可审的流程图
```

Skill 的本质是**把“怎么做一类事”沉淀成资产**。

**本项目可以蒸馏成哪些 Skill？**

| Skill | 职责 | 触发时机 |
|-------|------|----------|
| `explore-prd` | 读 PRD / 测试 / README，输出硬约束清单 | 阶段 1 |
| `design-spec` | 根据约束写 DESIGN.md（8 部分） | 阶段 2 |
| `mock-verify` | 对 DESIGN 做手推/脚本/桩函数验证 | 阶段 3 |
| `mod-impl` | 按 spec 实现单个 Python 模块 | 阶段 4 |
| `e2e-debug` | 端到端失败时定位根因 | 阶段 5 |
| `spec-check` | 收尾时对照 DESIGN 检查实现一致性 | 阶段 6 |

### 9.4 Gateway 路由思维：谁来处理这条消息？

TA-Claw 用 OpenClaw Gateway 做统一入口，自动把消息路由到对应 Skill：

```
学生提问 ──→ Gateway ──→ ta-answer Skill ──→ 生成回复
生成课件 ──→ Gateway ──→ ta-courseware Skill ──→ 调用 OpenMAIC
实验提醒 ──→ Gateway ──→ ta-experiment Skill ──→ Cron 触发
```

在 Claude Code 里，这种“路由”可以由人类完成：拿到一个任务后，先判断它属于哪个阶段、该用哪种 Agent，再发 prompt。不要所有事都丢给同一个通用会话。

### 9.5 本项目可以怎么用 Subagent 重做一遍？

假设重新开发无线通信仿真系统，可以这样拆分：

1. **Explore Agent**：只读 PRD.docx + public_tests/，输出“硬约束清单 + 验收门限 + 20 个测试覆盖矩阵”。
2. **Plan Agent**：根据硬约束写 DESIGN.md（含 #5 数据流推演、#6 预期风险）。
3. **Mock Verify Agent**：
   - 子任务 A：手推 QPSK 4 种比特对的 mod→demod 闭环
   - 子任务 B：3 行脚本算 preamble 自相关
   - 子任务 C：推演端到端比特数变化链
   - 三个子任务并行跑，汇总风险清单。
4. **Build Agent**：按已验证 DESIGN，并行生成 8 个 src/ 模块 + main.py。
5. **Verify Agent**：跑 22 个公开测试 + diff 文本 + 对照 DESIGN 检查接口一致性。

> 🔑 **核心认知**：Subagent 不是“多个 AI 一起写代码”，而是**把不同阶段的认知负载分给不同 AI，每个 AI 有明确的目标和边界**。人类负责编排（哪个 Agent 做什么、什么时候汇总结果）。

## 十、关键技巧清单（可直接套用）

### ✅ Do

1. **先结构，后代码**：一切动作围绕“目的”展开，不要在约束未稳定前就写实现
2. **先探索再规划**：用 Explore agent 读懂 PRD 和项目，别凭空让 AI 写
3. **写 DESIGN 规约**：把架构/接口/参数/风险写下来，人类审批后再实现（SDD）
4. **写代码前先 mock 验证设计**：手推/最小脚本/桩函数，验接口、帧结构、同步、端到端（Mock）
5. **先看公开测试再写代码**：让测试先失败（红），再写代码通过（绿）（TDD）
6. **用 Git Worktree 做物理隔离沙箱**：每个 phase/每个 Subagent 独立工作目录，防止污染主分支
7. **复杂任务拆 Subagent**：按功能模块/工作流阶段/独立文件拆分，给每个 Agent 明确边界
8. **并行生成独立文件**：一条消息多个 Write，大幅提速
9. **端到端先跑通**：先看宏观指标，再钻细节
10. **调试时缩范围**：单模块测试 → 打印中间数据 → 定位根因
11. **用测试做验收标准**：客观的 pass/fail 比"感觉对了"可靠
12. **缓存清理要记得**：改了代码不生效？先 `rm -rf __pycache__`
13. **收尾对照 spec**：代码/测试/报告都回溯对照 DESIGN
14. **记录 AI 使用日志**：哪些是 AI 写的、哪些是人改的、为什么采纳（也是交付物）
15. **先定义 MVP 再扩展**：第一版只验证核心假设，别一开始就求全
16. **把问题沉淀成资产**：每次 bug 都更新 Spec 或 Harness，避免重复踩坑

### ❌ Don't

1. **别跳过 DESIGN 直接写**：AI 自由发挥容易跑偏，返工成本更高
2. **别跳过 mock 验证设计**：设计 bug 留到运行时才发现，调试代价极高
3. **别让一个 AI 会话干所有事**：探索、规划、实现、验证混在一起，上下文会炸
4. **别让 AI 瞎改 bug**：不诊断根因就改，常常引入新 bug
5. **别在主分支上让 AI 直接实验**：先用 Worktree/临时分支，改坏了能丢
6. **别信"无噪声下也有错"的代码**：编解码模块无噪声必须完全可逆，有错就是 bug
7. **别用周期序列做同步前导**：伪峰值会让同步彻底失效
8. **别只看"跑通了"**：要核对输出内容是否真的正确（diff 比对）
9. **别让 AI 决定架构**：架构决策（用什么编码、什么调制）是人类的事
10. **别把文档当代码的附属品**：文档是规约信源，代码要对照它，不是反过来

## 十一、可复用的 Prompt 模式

### 模式 1：探索型（阶段 1）
```
探索 [项目路径]，我需要了解：
1. 读 PRD，提取硬约束、统一口径、验收入口
2. 列出所有文件和子目录
3. 总结项目目的和依赖
返回结构化总结，不要文件内容堆砌。
```

### 模式 2：规划型 / SDD（阶段 2）
```
基于探索结果，写 DESIGN.md，包含 6 个部分（参考 2.3 Spec 详解）：
1. 系统概述（一句话讲清做什么）
2. 系统链路 + 模块设计 + 关键参数（含数据流图、接口签名、参数表）
3. 数据流推演（推演比特数变化链，验收发对齐）
4. 预期风险（≥5 条，现象+根因+缓解）
5. 文件结构（目录树）
6. 运行方式 + 验收标准（CLI 命令 + 通过门限）
```

### 模式 3：Mock 验证型（阶段 3）★
```
我的 DESIGN 里 QPSK 映射规则是：00→(1+1j)/√2, 01→(-1+1j)/√2, 11→(-1-1j)/√2, 10→(1-1j)/√2。
请帮我手推：给定一个解调后的复数点，应该用什么规则判回比特对？
要求：列出 4 种情况，说明 b0/b1 分别由实部还是虚部决定，保证闭环可逆。
不要写代码，只要推理。
```
```
我的 DESIGN 用 [1+1j,-1+1j,-1-1j,1-1j]*7 的前 25 个符号做同步前导。
请写一个 5 行以内的脚本，算这个前导的自相关，判断它是否适合做同步（是否有伪峰值）。
只验证这一个假设，不要写完整模块。
```

### 模式 4：调试型（阶段 5）
```
[模块] 在 [条件] 下出现 [现象]。
写一段调试脚本：
1. 单独测试该模块
2. 逐元素打印发送 vs 接收
3. 定位第一个出错的位置
不要直接改代码，先帮我看清错在哪。
```

### 模式 5：验证型 / 对照 spec（阶段 6）
```
运行 [测试命令]，报告结果。
然后用 [diff/比对] 确认 [输出] 与 [预期] 一致。
最后检查代码实现是否与 DESIGN.md 的接口定义一致，列出不一致项。
```

### 模式 6：Subagent 拆分型（跨阶段）
```
我要实现 [复杂功能]，请把它拆成 3 个可独立验证的子任务：
1. [子任务 A：探索/读代码]
2. [子任务 B：设计/写 spec]
3. [子任务 C：实现/验证]
每个子任务给出：目标、输入、输出、验收标准。先不生成最终代码，只确认拆分是否合理。
```

> 这六种 prompt 的共同点：**让 AI 验证设计和意图，而不是放任 AI 自由写代码**。这是 vibe coding 在 AI 协作中的具体落地。

## 十二、三种流程对比与本项目实证

### 对比：三种 AI 开发流程

| 维度 | 纯 vibe（凭感觉写） | 边写边调（我们部分采用） | 设计验证驱动（规范） |
|------|---------------------|------------------------|---------------------|
| 代码何时写 | 很早就写 | 较早写 | DESIGN 验证稳定后才写 |
| bug 何时发现 | 运行时 | 运行时 | 设计阶段（mock） |
| bug 性质 | 代码 bug | 常被误当"代码 bug" | 暴露为"设计 bug" |
| 调试成本 | 高 | 高 | 低（改文档即可） |
| DESIGN 的作用 | 无 | 事后补的文档 | 事前的施工图 |
| MOCK_REPORT | 无 | 事后编的"测试场景" | 事前真实的设计验证记录 |
| 工作隔离 | 无 | 直接在主分支改 | Git Worktree / 临时分支物理隔离 |
| 任务拆分 | 单会话全包 | 基本不分 | 按模块/阶段拆 Subagent，并行验证 |
| 框架覆盖 | 无 | 部分 | Vibe→Spec→Harness + SDD+Mock+TDD+Subagent+Worktree 全覆盖 |
| 适合场景 | 玩具/探索 | 简单项目 | 有 PRD、要交付验收的项目 |

### 本项目工作流实证（时间线）

按 **Vibe → Spec → Harness** 三层重新归类时间消耗：

| 阶段 | 框架层 | 耗时占比 | 人类贡献 | AI 贡献 | 方法论 |
|------|--------|---------|---------|---------|--------|
| 0 环境准备 | Vibe | 5% | 装环境 | — | — |
| 1 需探索 | Vibe | 10% | 审阅理解 | 并行探索 7 个文件 | SDD 喂料 |
| 2 方案规划 | Spec | 15% | 审批+定架构 | 起草 DESIGN | SDD |
| 3 mock 验证 | Harness | 0%（跳过了） | — | — | Mock（该做没做） |
| 4 并行实现 | Harness | 20% | 观察 | 批量生成 13 个文件 | TDD（顺序反了） |
| 5 运行调试 | Harness | 40% | 诊断 2 个根因 bug | 写调试脚本+修复 | — |
| 6 验证收尾 | Harness | 10% | 复核输出 | 跑测试 | 对照 spec |

**关键观察**：
1. **Vibe + Spec（探索+规约）只占 30%**，Harness 阶段（执行+验证）占了 70%——AI 把"写代码"压缩到 20%，但"保证代码正确"无法压缩，更依赖人类诊断能力。
2. 我们跳过了阶段 3（Harness 中的 Mock 验证），导致两个设计 bug 流到运行时，白白多花了调试时间。**如果做了 mock，调试占比能从 40% 降到 10% 左右**——这就是"先 design 再对 design 验证"的量化价值。
3. 规范流程下合理的时间分配应该是 **Vibe 20% + Spec 20% + Harness 60%**，其中 Harness 内 Mock 验证 ≥ 10%、TDD 实现 30%、Debug 10%、Verify 10%。本项目缺了 Mock 验证那一环，导致 Harness 内部严重失衡。

---

## 十三、四条铁律

1. **意图归人类，执行归 AI**——你说"要什么"，AI 管"怎么写"。（SDD 的本质）
2. **结构先于代码，隔离先于污染，拆分先于混杂**——先写 spec 再写代码，用 Worktree 隔离实验，用 Subagent 拆分复杂任务。（SwiftClaw “先结构，后代码”的本质）
3. **规划先于实现，诊断先于修复，验证先于代码**——动手前想清楚，改 bug 前看明白，写代码前先 mock 验设计。（Mock 的本质）
4. **用客观验证收尾，规约是唯一信源**——测试通过才算完，代码/测试/报告都对照 DESIGN。（TDD + SDD 的本质）

掌握这四条，AI 就是你最高效的编程搭档；违背任何一条，AI 就会变成制造返工的源头。
