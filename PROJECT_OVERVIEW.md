# 📡 无线通信基带仿真系统 — 项目介绍

> 一句话概括：把一段中文文本"发"过一条模拟无线信道，再在接收端"原样"还原回来——中间完整跑通一条真实通信系统的基带处理链。

---

## 一、这个项目能干什么？

本项目实现了一个**端到端的无线通信基带仿真系统**。它模拟了真实无线通信（比如 4G/5G、Wi-Fi）中从"按下发送"到"对方收到"之间发生的所有底层信号处理过程。

具体来说，它能做这些事：

| 能力 | 说明 |
|------|------|
| 📝 **文本传输** | 把任意 UTF-8 中文文本编码成比特流，传输后在接收端一字不差地恢复 |
| 🌊 **完整通信链路** | 跑通信源编码 → 加扰 → 信道编码 → 组帧 → QPSK 调制 → AWGN 信道 → 同步 → 解调 → 解码 → 解扰 → 信源解码 |
| 📡 **QPSK 调制仿真** | Gray 编码 QPSK 星座映射，单位功率，支持星座图可视化 |
| 🔊 **信道噪声模拟** | 模拟加性高斯白噪声（AWGN）信道，可调信噪比（SNR） |
| 🎯 **帧同步** | 基于互相关峰值检测，在噪声中找到数据帧的起始位置 |
| 🛡️ **纠错编码** | (3,1) 重复码前向纠错（FEC），抵抗信道噪声 |
| 📊 **性能评估** | 自动计算 BER（误码率）、FER（误帧率）、文本匹配率等指标 |
| 📈 **结果可视化** | 生成星座图、BER 曲线、同步峰值图 |

**典型应用场景：**
- 课程教学演示（理解无线通信各环节如何协作）
- 通信算法原型验证（测试不同 SNR 下的传输可靠性）
- 学习数字调制、信道编码、同步等核心概念

## 二、设计链路总览

整个系统遵循"**发射端处理 → 信道传输 → 接收端处理**"的经典三段式架构。数据像流水线一样逐级流转：

```
┌─────────────────────────── 发射端 (TX) ───────────────────────────┐
│                                                                    │
│  中文文本                                                          │
│     │                                                              │
│     ▼  ① Source Encode（信源编码）                                 │
│  比特流（UTF-8 → 字节 → 比特，8位对齐）                            │
│     │                                                              │
│     ▼  ② Scramble/Encrypt（加扰）                                  │
│  加扰比特流（PRNG 伪随机序列 XOR）                                 │
│     │                                                              │
│     ▼  ③ Channel Encode（信道编码）                                │
│  纠错编码比特流（(3,1) 重复码，每比特重复3次）                     │
│     │                                                              │
│     ▼  ④ Frame Build（组帧）                                       │
│  完整帧（Preamble + Length + Payload + CRC-16）                    │
│     │                                                              │
│     ▼  ⑤ QPSK Modulate（调制）                                     │
│  复数符号序列（Gray 编码 QPSK，单位功率）                          │
│                                                                    │
└──────────────────────────────┬─────────────────────────────────────┘
                               │
                               ▼  ⑥ AWGN Channel（信道）
                        加噪后的复数信号
                               │
┌──────────────────────────────┴─────────────────────────────────────┐
│                              接收端 (RX)                            │
│                                                                    │
│     ▼  ⑦ Synchronization（同步）                                   │
│  找到帧起始位置（互相关峰值检测）                                  │
│     │                                                              │
│     ▼  ⑧ QPSK Demodulate（解调）                                   │
│  恢复比特流（按象限判决）                                          │
│     │                                                              │
│     ▼  ⑨ Frame Parse（解析帧）                                     │
│  提取 Payload（依据 Length 字段去掉 padding）                      │
│     │                                                              │
│     ▼  ⑩ Channel Decode（信道解码）                                │
│  多数投票解码（3中取2，纠正单比特错误）                            │
│     │                                                              │
│     ▼  ⑪ Descramble（解扰）                                        │
│  还原加扰前的比特（同种子 XOR 自逆）                               │
│     │                                                              │
│     ▼  ⑫ Source Decode（信源解码）                                 │
│  恢复中文文本（比特 → 字节 → UTF-8）                               │
│                                                                    │
└────────────────────────────────────────────────────────────────────┘
                               │
                               ▼
                         ⑬ Metrics（指标计算）
                  BER / FER / text_match_rate / 图表
```

> **设计哲学**：每一级都是可逆的——发射端做什么，接收端就"反着做"什么。信道是唯一会"破坏"数据的地方（加噪声），而信道编码 + 同步就是为了在噪声中把数据救回来。

## 三、各模块功能详解

源代码全部在 `src/` 目录下，每个模块对应通信链的一个环节：

### 1. `src/source.py` — 信源编码 ①⑫
**功能**：文本与比特流的相互转换。
- `source_encode(text)`：中文文本 → UTF-8 字节 → 比特列表（MSB 在前，8 位对齐）
- `source_decode(bits)`：逆过程，比特 → 字节 → UTF-8 文本
- **设计要点**：保证比特数是 8 的倍数，这样能完整还原任意 UTF-8 字符（中文通常占 3 字节）。

### 2. `src/crypto.py` — 加扰/解扰 ②⑪
**功能**：比特级可逆加扰，让数据呈现"随机性"（避免长串 0/1 影响同步和时钟恢复）。
- `scramble(bits, seed)`：用 `numpy.random.default_rng(seed)` 生成伪随机比特，与输入 XOR
- `descramble(bits, seed)`：相同种子再 XOR 一次即还原（XOR 自逆）
- **设计要点**：收发两端必须用相同种子，否则无法还原。

### 3. `src/channel_coding.py` — 信道编码 ③⑩
**功能**：前向纠错（FEC），抵抗信道噪声造成的比特错误。
- `channel_encode(bits)`：(3,1) 重复码，每个比特重复 3 次
- `channel_decode(bits)`：3 比特一组多数投票（3 中取 2）
- **设计要点**：编码率 1/3，牺牲带宽换可靠性。无噪声时完全可逆；有噪声时能纠正单个比特错误（3 个里错 1 个仍可救回）。在 SNR=12dB 下绰绰有余。

### 4. `src/framing.py` — 组帧/解析 ④⑨
**功能**：给数据加上"信封"，让接收端知道帧从哪开始、有多长、是否完整。
- `build_frame(payload)`：组装帧结构
- `parse_frame(frame)`：从帧比特中提取各字段

**帧结构设计：**

| 字段 | 长度 | 作用 |
|------|------|------|
| Preamble（前导） | 50 bit = 25 个 QPSK 符号 | 伪随机已知序列，用于同步 |
| Length（长度） | 16 bit | 标明 Payload 的比特数（最大 65535） |
| Payload（载荷） | 可变 | 实际要传的数据 |
| Checksum（校验） | 16 bit | CRC-16，验证帧完整性 |

- **设计要点**：Preamble 用独立种子（seed=42）生成伪随机 QPSK 符号，保证良好的自相关特性（尖锐单峰），避免同步时出现伪峰值。Length 字段让接收端能精确去掉调制时补的 padding。

### 5. `src/modulation.py` — QPSK 调制/解调 ⑤⑧
**功能**：把比特"画"到复平面星座图上，再从星座点读回比特。
- `qpsk_modulate(bits)`：比特对 → 复数符号
- `qpsk_demodulate(symbols)`：复数符号 → 比特对

**Gray 编码 QPSK 星座映射（PRD 规定）：**

| 比特对 | 象限 | 复数符号 | 实部 | 虚部 |
|--------|------|----------|------|------|
| 00 | Q1 | (1+1j)/√2 | + | + |
| 01 | Q2 | (-1+1j)/√2 | − | + |
| 11 | Q3 | (-1-1j)/√2 | − | − |
| 10 | Q4 | (1-1j)/√2 | + | − |

- **设计要点**：Gray 编码使相邻星座点只差 1 个比特（噪声导致判错象限时，只错 1 bit 而非 2 bit）。归一化系数 1/√2 保证符号平均功率为 1。奇数比特自动补 0。
- **解调判决规则**：`b0` 由虚部符号决定（虚部<0 → 1），`b1` 由实部符号决定（实部<0 → 1）。

### 6. `src/channel.py` — AWGN 信道 ⑥
**功能**：模拟真实无线信道的噪声，是整个系统中唯一"破坏"数据的环节。
- `awgn(symbols, snr_db, seed)`：给信号叠加复高斯噪声
- **设计要点**：
  - 根据信号功率和 SNR 计算噪声方差：`noise_power = signal_power / 10^(snr_db/10)`
  - 复噪声实部、虚部各分一半方差
  - 固定随机种子保证结果可复现（相同输入 → 相同输出）

### 7. `src/synchronization.py` — 同步 ⑦
**功能**：在接收信号中找到数据帧的起始位置。
- `synchronize(received, preamble)`：返回帧起始索引
- **设计要点**：用**互相关（匹配滤波）**——把接收信号与已知 Preamble 滑动做内积，内积幅值最大的位置就是帧起点。Preamble 越随机，峰值越尖锐，同步越准。

### 8. `src/metrics.py` — 指标计算 ⑬
**功能**：量化系统传输质量。
- `compute_ber(original, received)`：比特错误率 = 错误比特数 / 总比特数
- `compute_text_match_rate(orig, recv)`：文本字符匹配率
- `compute_checksum_pass(...)`：CRC 校验是否通过

## 四、项目入口与使用方法

### 4.1 入口文件：`main.py`

`main.py` 是整个项目的**统一 CLI 入口**。它把上面 8 个模块按设计链路顺序串联起来，一条命令完成"文本→发射→信道→接收→文本"的全流程。

**它做了什么：**
1. 读取输入文本文件
2. 依次调用发射端模块（信源编码 → 加扰 → 信道编码 → 组帧 → 调制）
3. 过 AWGN 信道
4. 依次调用接收端模块（同步 → 解调 → 解析 → 解码 → 解扰 → 信源解码）
5. 写出恢复文本、性能指标、可视化图表

### 4.2 环境准备（首次使用）

```bash
# 1. 进入项目目录
cd wireless-final-project-template

# 2. 创建并激活 conda 虚拟环境
conda create -n wireless-final python=3.11 -y
conda activate wireless-final

# 3. 安装依赖（numpy / scipy / matplotlib / pytest）
pip install -r requirements.txt
```

### 4.3 运行项目（核心命令）

```bash
python main.py --input Test.txt --output results/received.txt \
               --snr 12 --seed 2026 --mod qpsk --channel awgn
```

**参数说明：**

| 参数 | 含义 | 示例值 |
|------|------|--------|
| `--input` | 输入文本文件路径 | `Test.txt` |
| `--output` | 输出恢复文本路径 | `results/received.txt` |
| `--snr` | 信噪比（dB），越大噪声越小 | `12` |
| `--seed` | 随机种子，保证可复现 | `2026` |
| `--mod` | 调制方式 | `qpsk` |
| `--channel` | 信道模型 | `awgn` |

### 4.4 运行结果

运行后会在 `results/` 目录生成：

| 文件 | 内容 |
|------|------|
| `received.txt` | 接收端恢复的文本（SNR=12dB 时与原文件完全一致） |
| `metrics.json` | 性能指标（BER、FER、text_match_rate、sync_start_index 等） |
| `constellation.png` | QPSK 接收星座图（看噪声扩散程度） |
| `ber_curve.png` | BER 随 SNR 变化曲线（理论值 vs 实测值） |
| `sync_peak.png` | 同步互相关峰值图（看帧起始检测） |

**成功运行的输出示例：**
```
传输完成！
  SNR: 12.0 dB
  Payload bits: 1544
  BER: 0.000000
  FER: 0.0
  Text match rate: 1.0
  Sync start index: 0
```

### 4.5 运行测试

```bash
# 运行全部 22 个公开测试用例
PYTHONPATH=. pytest public_tests/ -v
```

## 七、关键设计决策与背后的思考

### 1. 为什么用 Gray 编码？
普通二进制编码下，相邻星座点可能差 2 个比特（如 00↔11）。噪声导致判到相邻象限时，会错 2 bit。Gray 编码让相邻点只差 1 bit，**噪声造成的错误比特数减半**，BER 性能更好。

### 2. 为什么用 (3,1) 重复码？
它简单、可靠、无噪声时完全可逆。在 SNR=12dB 的高信噪比场景下，单比特错误概率本就很低，重复码的多数投票足以把零星错误纠正过来，保证 text_match_rate=1.0。若要追求更高编码效率，可换汉明码或卷积码。

### 3. 为什么 Preamble 要用伪随机序列？
早期版本用了周期性序列（`[1+1j, -1+1j, ...]` 循环），结果同步时**到处都是伪峰值**——因为周期序列和自身的循环移位高度相关，导致找不到真正的帧起点。改用伪随机序列后，自相关呈现**尖锐单峰**，同步立刻精准定位。

### 4. 为什么收发两端要用相同随机种子？
加扰、AWGN 信道都依赖伪随机数。加扰的种子必须在收发两端一致（否则解扰失败）；AWGN 的种子只影响噪声样本，但固定种子保证**实验可复现**——同样参数每次跑出一样的结果，便于调试和验证。

## 八、快速验证

想确认项目跑通了？一行命令：

```bash
conda activate wireless-final && \
python main.py --input Test.txt --output results/received.txt \
               --snr 12 --seed 2026 --mod qpsk --channel awgn && \
diff Test.txt results/received.txt && echo "✅ 文本完全恢复！"
```

看到 `✅ 文本完全恢复！` 就说明整条通信链路工作正常。
